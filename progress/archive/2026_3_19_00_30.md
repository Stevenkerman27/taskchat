✦ 我已经检查了这组代码，并且修复了其中隐藏的几个比较关键的逻辑缺陷和异常处理问题：


   1. `execute_pending_tools` 异常处理越界导致的崩溃 (BUG) :
      - 原情况： 之前在执行工具时，如果某个工具内部报错（例如抛出异常），外层的 try-except 会捕获异常并返回 [{"error": "工具执行失败..."}]，由于缺少 function_name 键，会导致 GUI  
        在遍历执行结果时抛出 KeyError: 'function_name'。
      - 修复： 将异常捕获缩小到单个工具循环内部。如果工具执行失败，会记录错误信息作为该工具的 result 反馈，并将工具标记为已执行。这样不仅能防止 GUI
        崩溃，还能让大模型看到报错信息，从而有可能自我修正参数并重试。


   2. `cancel_tool_calls` 上下文状态破坏 (BUG) :
      - 原情况： 取消工具调用时，代码固定地只是 pop() 一条助手消息和一条用户消息。如果在多次多步工具调用途中（例如 助手调用工具1 -> 用户返回工具1结果 ->
        助手接着调用工具2），用户此时点击“取消”，原逻辑会导致上下文残留未能配对的 tool_result，使得历史记录错乱，API 直接报错 "messages follow incorrect sequence"。
      - 修复： 改为使用 while
        循环回退历史记录，从末尾开始删除，直到遇到最后一个不带工具调用的正常助手消息（或者系统消息）为止。这确保了我们能干净地“撤销”本轮导致工具链产生的所有未完成消息状态。       


   3. `is_in_reasoning_mode` 导致永远无法保存聊天记录 (逻辑漏洞) :
      - 原情况： 之前在判断能否保存记录时，代码只要检测到历史最后一条消息的 metadata 里带有 "reasoning"，就会判定为处于不完整状态并拒绝保存（提示
        "无法在思维链中途保存"）。但实际上一旦某次对话使用了思维链，消息完整生成后也会带这个字段，这会导致之后无论何时都永远无法保存。
      - 修复： 移除了对已完成消息带有 reasoning 字段的误判。现在只检查是否处于真正的 tool_call_mode 或有未执行完毕的 pending_tools，确保正常的对话都可以顺利保存。


   4. GUI 加载记录时针对字典对象解析导致无内容的隐患 (BUG) :
      - 原情况： 在 load_chat_context 的 UI 预览消息时，代码假设反序列化后的 first_part 必定拥有 .content 属性（hasattr(first_part, 'content')）。但反序列化时有些对象会作为基础
        Python 字典存在（没有经过完整的 Pydantic 解析）。此时走到 else 分支直接使用了 str(first_part)，让用户在预览里看到一串纯字典字符串。
      - 修复： 添加了针对 isinstance(..., dict) 并且含有 'content' 键的判断分支，能够正确提取出反序列化后的字典文字内容。
