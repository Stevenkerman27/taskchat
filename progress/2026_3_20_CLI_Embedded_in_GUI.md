# 2026.3.20 CLI终端集成到GUI及子进程通信编码修复

## 决策意图
1. **解决子进程中文字符乱码问题**：在彻底转为 GUI 包装器（双进程架构）后，CLI 子进程的输出通过管道传给 GUI 父进程。在 Windows 操作系统中，系统默认的终端编码通常为 GBK (`cp936`)。这导致 CLI 在打印包含中文字符的 JSON 状态或消息时，管道传输的字节流与预期编码不符，导致 GUI 端的 `json.loads` 发生 `JSONDecodeError`，从而在界面上仅能显示 `[CLI RAW]` 后接一串乱码。为此，在 `chat_gui_v2.py` 启动 `subprocess.Popen` 时，通过强行注入 `PYTHONIOENCODING="utf-8"` 和 `PYTHONUTF8="1"` 环境变量，同时给解释器附加 `-X utf8` 标志，确保了子进程的标准输入输出被强制锁定在 UTF-8 编码，消除了环境差异导致的通讯崩溃。
2. **GUI内嵌极客控制台（双输入框设计）**：为了满足高级用户对快捷指令的需求，同时不破坏普通用户使用主聊天输入框输入自然语言的体验，在 GUI 中单独划出了一块 `>_ CLI Command` 输入区域。在协议层面，我们在 JSON 模式的指令集中新增了 `"cmd": "raw"` 路由。这使得 GUI 可以直接将用户输入的类似 `/provider openai` 的字符串扔给 CLI 进程。CLI 进程会将其转发给本来就存在的 `process_text_command()` 进行解析。这种设计**坚决捍卫了 DRY 原则**，前端不仅一行正则都不需要写，而且凭借我们现有的“状态广播 (state event)”机制，用户敲入命令后导致的状态变更（如切换了模型），会自动通过 JSON 流传回给 GUI，并瞬间更新顶部的下拉菜单，形成了极为优雅的数据双向同步。

## 防御性记录
- **环境隔离造成的“伪假死”**：由于新版 GUI 只依赖标准库 `tkinter`，即使子进程所需的重要库（如 `openai` SDK）缺失，GUI 依然能顺利拉起。这导致了开发者初期对环境配置产生了误判（以为界面出来了就说明库装好了）。随后我们在日志捕获中明确打印出了 CLI 在初始化 provider 时的 `ImportError` 警告，这提醒了我们：在多进程架构下，排查业务错误必须时刻关注子进程返回的 stderr 和初始化事件，不可被前端的状态所迷惑。
- **关于前端解析命令的否决方案**：曾考虑过在 GUI 主聊天框中直接拦截以 `/` 开头的文本，并在前端解析出动作转为结构化 JSON 指令发送。此方案被否决。因为前端不应该承担任何业务逻辑的“猜测”工作，一旦 CLI 端命令更新或变动，GUI 代码需要同步维护，严重违反开闭原则。最终采纳了开放 `raw` 接口将解析权完全丢回给 CLI 内部的方案。

## 验证依据
- **中文字符跨进程边界测试**：执行包含中文字符的回复请求和带有中文命名的历史上下文拉取，验证 GUI 能正确反序列化 JSON，并使用 Tkinter 正常渲染，无 `[CLI RAW]` 报错。
- **指令与 UI 的响应绑定测试**：在专用命令行文本框输入 `/provider openai` 或 `/option temperature 0.8` 并回车，验证不仅聊天输出区给出了正确的反馈（通过事件流回传），且上方的 Provider 下拉框及 Advanced Options 中的数值都被自动更新。证明指令下发和状态回收的闭环已完全跑通。
